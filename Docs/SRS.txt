2.2 Product Functions
The Convera system will provide the following main functions:
• Secure User Authentication using Email/Password and OTP verification.
• Event Discovery based on location, date, price, and eligibility.
• Accommodation Search (apartments, hotels, stays) similar to Airbnb interface.
• Advanced Filtering (price, rating, distance, availability).
• Property Details Viewing including images, reviews, and amenities.
• Real-Time Booking System with availability updates.
• Secure Online Payment integration.
• Reservation Management (view, modify, cancel bookings).
• User Profile Management (personal data, booking history).
• Host-User Chat System for direct communication.
• Reviews & Ratings System for properties.
• Automated Email Notifications (booking confirmation, updates).
• Admin Dashboard for managing users, listings, and events.

4. Non-Functional Requirements (Convera)

4.1 Performance Requirements
• The system should support at least 1,500 concurrent users without performance degradation.
• Response time for user actions (search, booking, chat, filtering) should not exceed 500
milliseconds under normal load.
• The system should handle up to 15,000 daily transactions (bookings, payments, messages).
• Search results (events/properties) should load within 2 seconds maximum.
• Real-time features (chat, availability updates) should have latency ‫أقل من‬1 second.

4.2 Reliability
• In case of system failure, automatic recovery should occur within 2 minutes.
• Data must be backed up automatically every 4–6 hours.
• Booking transactions must be atomic (either fully completed or fully canceled).
• In case of network failure, pending transactions (payments/bookings) should be queued and
retried.
• Message delivery in chat system must be reliable with retry mechanisms.

4.3 Availability
• The system must maintain 99.9% uptime excluding scheduled maintenance.
• Critical services (authentication, booking, payment) must always be available.
• Maintenance periods should be announced in advance.

4.4 Scalability
• The system should support horizontal scaling (adding servers).
• The system should support expansion to multiple cities and countries.
• The system should handle increasing number of:
•
•
•

Users
Properties
Events

• The system should support integration with multiple payment gateways depending on region.
• The system should scale chat services separately (microservices approach).
